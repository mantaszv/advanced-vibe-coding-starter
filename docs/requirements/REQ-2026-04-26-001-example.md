# REQ-2026-04-26-001. Pavyzdinis PRD: vartotojo profilio puslapis

> **Pastaba.** Tai pavyzdinis PRD, demonstruojantis CWK 4 etapų pipeline output formatą. Realų PRD sugeneruos `/create-prd "..."` komanda.

## 1. Apžvalga

Pridėti vartotojo profilio puslapį. Vartotojas jame mato ir redaguoja savo asmeninius duomenis: vardą, el. paštą, profilio nuotrauką.

## 2. Tikslai

- Vartotojas mato savo asmens duomenis viename puslapyje.
- Vartotojas gali atnaujinti vardą ir el. paštą su validacija.
- Profilio nuotrauka įkeliama per Supabase Storage.

## 3. User Stories

- *Kaip prisijungęs vartotojas*, noriu pamatyti savo profilio info. Tada žinosiu, kokius duomenis sistema apie mane turi.
- *Kaip vartotojas*, noriu redaguoti el. paštą. Tada pranešimai keliaus į teisingą adresą.

## 4. Funkciniai reikalavimai

- **FR-1.** Maršrutas `/profile` pasiekiamas tik prisijungusiems vartotojams (auth middleware).
- **FR-2.** Puslapyje rodomi: vardas (tik skaitymui iš Supabase auth), el. paštas (galima keisti), profilio nuotrauka (galima keisti).
- **FR-3.** El. Pašto pakeitimas reikalauja patvirtinimo per email (Supabase auth flow).
- **FR-4.** Nuotraukos įkėlimas ribojamas iki 2MB. Leistini tipai: PNG, JPEG.

## 5. Ne tikslai (Non-Goals)

- Slaptažodžio keitimas. Tai atskiras flow'as `/account/security`.
- Dviejų faktorių autentifikacija.
- Prenumeratos (subscription) valdymas.

## 6. Dizaino svarstymai

- shadcn/ui `<Card>` ir `<Form>` komponentai.
- Optimistic UI atnaujinimai email keitimui.
- Toast pranešimai apie sėkmę ir klaidas.

## 7. Techniniai svarstymai

- Supabase RLS politikos. Vartotojas mato ir keičia tik savo duomenis (`auth.uid() = user_id`).
- Server Action `updateProfile` su Zod validacija.
- Storage bucket'as `avatars/`. Public read, authenticated write.

## 8. Orchestration Hints

| Komponentas | Kas reikalinga |
|---|---|
| **MCP serveriai** | Supabase (DB queries, RLS), Context7 (Next.js Server Actions dokumentacija), Playwright (E2E) |
| **Agentai** | `Explore` (rasti esamus auth pattern'us), `backend-architect` (Server Action), `frontend-architect` (Form komponentas) |
| **PRE Guards** | `security-guard` (RLS validacija), `risk-assessor` (auth flow rizika) |
| **POST Guards** | `language-guard` (LT UI tekstai), `file-size-guard` (komponentų dydis), `test-coverage-guard` (bent 80% coverage) |
| **Skills** | `superpowers:test-driven-development` (optional, jei plugin'as įdiegtas) |
| **Verify** | `npm run build && npm run lint && npx vitest run && npx playwright test` |

## 9. Rizikos vertinimas

| Rizika | Tikimybė | Poveikis | Mitigacija |
|---|---|---|---|
| El. Pašto pakeitimas be patvirtinimo (security gap) | Žema | Aukštas | Supabase native confirm flow ir automatinis logout siunčiančiam vartotojui |
| RLS politikos klaida. Vartotojas mato kitų duomenis | Vidutinė | Kritinis | E2E testas su dviem vartotojais. `security-guard` PRE gate |
| Nuotraukų storage bucket'as per atviras | Žema | Vidutinis | RLS storage politikos. Write tik `auth.uid() = user_id` |
| Form'ų server action neturi rate limiting'o | Vidutinė | Vidutinis | Pridėti rate-limit middleware (10 užklausų per min) |

## 10. Sėkmės metrikos

- 100% vartotojų pasiekia `/profile` po prisijungimo.
- Per pirmus 30 dienų 0 RLS pažeidimų production'e.
- E2E testas praeina greičiau nei per 5 sekundes.
- Nuotraukos įkėlimo p95 latencija mažesnė nei 2 sekundės.
